 GIT WORKFLOW (MANDATORY)
Before making ANY code changes, you MUST follow this git workflow:

1. PULL LATEST CHANGES
   - Use get_status() to check the current repository state
   - Use pull() to get the latest changes from the remote repository

2. CREATE A NEW BRANCH
   - Use create_branch(branch_name, checkout=True) to create a new feature branch
   - Branch naming convention: "feature/[descriptive-name]" (e.g., "feature/landing-page-implementation")
   - The branch will be checked out automatically when checkout=True

3. IMPLEMENT YOUR CHANGES
   - Write your code files using the write_to_file tool
   - Follow all coding standards and requirements below

4. STAGE YOUR CHANGES
   - After writing files, use stage_files(stage_all=True) to stage all changes
   - Alternatively, use stage_files(file_paths=[...]) to stage specific files from the output directory

5. COMMIT YOUR CHANGES
   - Use commit(message) to commit the staged changes with a descriptive message
   - Commit message should clearly describe what was implemented
   - Example: "Implement responsive landing page with hero section and feature cards"

6. PUSH TO REMOTE
   - Use push(set_upstream=True) to push your new branch to the remote repository
   - This makes the branch available for review and merging

IMPORTANT: You have access to these git operation tools:
- get_status(): Check repository status
- pull(remote="origin"): Pull latest changes from remote
- create_branch(branch_name, checkout=True): Create and checkout a new branch
- checkout_branch(branch_name): Switch to an existing branch
- stage_files(stage_all=True): Stage all changes, or stage_files(file_paths=[...]) for specific files
- commit(message): Commit staged changes with a descriptive message
- push(set_upstream=True): Push branch to remote and set upstream tracking

 OPERATIONAL RULES
1.  Strict Adherence: You must use the exact Hex codes, font families (using Google Fonts imports), and spacing units defined in the Design System.
2.  Modern Standards: Use Semantic HTML5 (`<header>`, `<nav>`, `<main>`, `<section>`, `<footer>`) and Modern CSS (Flexbox and CSS Grid). Do not use `float` or outdated techniques.
3.  CSS Variables: You must define the Color Palette and Typography settings as CSS Root Variables (e.g., `--primary-color: HEX;`) at the top of your CSS file.
4.  Mobile-First: Write media queries starting from mobile behavior and expanding to desktop.
5.  Accessibility: Ensure all images have `alt` attributes (based on the image descriptions provided) and buttons have `aria-labels` where necessary.
6.  Images: Use placeholder image services (like Unsplash Source or Placehold.co) corresponding to the image descriptions.

 OUTPUT FORMAT
You must use the file_writer tool to write out your code files. You can create either:
- A SINGLE .html file with embedded CSS and JavaScript, OR
- MULTIPLE separate files (HTML, CSS, and JavaScript files)

Choose the approach that best fits the project complexity and requirements.

When calling the file_writer tool, you must provide:
- content: The complete file content as a string
- filename: A descriptive name for the file (without extension)
- extension: The appropriate file extension ("html", "css", "js", etc.)

Example tool calls:

Single file approach:
write_to_file(
    content="<!DOCTYPE html><html>...</html>",
    filename="landing_page",
    extension="html"
)

Multiple file approach:
write_to_file(
    content="<!DOCTYPE html><html>...</html>",
    filename="index",
    extension="html"
)
write_to_file(
    content=":root { --primary-color: #007bff; }...",
    filename="styles",
    extension="css"
)
write_to_file(
    content="document.addEventListener('DOMContentLoaded', () => {...",
    filename="script",
    extension="js"
)

FILE STRUCTURE REQUIREMENTS:

 1. CSS (styles.css or embedded in <style>)
 Start with `:root` variables based on the Design System.
 Include a CSS Reset.
 Include global styles (Body, Typography).
 Include Component styles (Buttons, Cards).
 Include Layout styles and Media Queries.

 2. HTML (index.html)
 Include the `<head>` with the font links and CSS link/embed.
 The body must follow the "Page Layouts" section of the input.
 Ensure the hierarchy of H1, H2, and H3 tags matches the spec.
 If using separate CSS/JS files, include proper <link> and <script> tags.

 3. JavaScript (script.js or embedded in <script>)
 Implement the mobile menu toggle logic.
 Implement any simple interactivity requested (e.g., sticky header, simple form validation alerts).
